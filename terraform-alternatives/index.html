If you are an infrastructure engineer evaluating options beyond Terraform, you are likely looking for different approaches to infrastructure as code. Terraform has been the dominant tool for managing cloud resources declaratively, but it is not the only choice. Depending on your team’s language preferences, cloud provider lock-in tolerance, or desired level of abstraction, alternatives such as Pulumi, AWS CloudFormation, and Crossplane may offer advantages. This article compares those three tools so you can decide which fits your infrastructure workflows.

alternative

Pulumi

. Pulumi lets you define infrastructure using general-purpose programming languages like TypeScript, Python, Go, and C# instead of a custom DSL. This means you can use familiar coding patterns, test frameworks, and IDE features (autocomplete, refactoring) directly in your IaC code. Pulumi supports multiple clouds and offers state management similar to Terraform, but with a stronger emphasis on software engineering practices for infrastructure.

alternative

AWS CloudFormation

. CloudFormation is the native infrastructure as code service for AWS, using YAML or JSON templates to define resources in a stack. It is deeply integrated with AWS features such as Service Catalog, StackSets, and drift detection, and it does not require a separate state file because AWS manages the stack state on your behalf. However, it is tied to AWS only, so it is not suitable for multi-cloud or hybrid environments.

alternative

Crossplane

. Crossplane takes a Kubernetes-native approach by extending Kubernetes custom resources to provision and manage cloud infrastructure. It does not rely on a separate CLI or state backend; instead, it uses the Kubernetes control plane to reconcile desired state with actual cloud resources. Crossplane is ideal for teams that already run Kubernetes and want a consistent way to manage both containerized applications and the underlying cloud infrastructure from the same API.

How they compare

All three tools provide infrastructure as code but differ in their ecosystem and abstraction model. Pulumi offers the most flexibility with real programming languages and multi-cloud support, making it a strong alternative for teams that value code reuse and testing. AWS CloudFormation trades that flexibility for deep AWS integration and built-in state management, best suited for AWS-only shops. Crossplane brings infrastructure management into Kubernetes, which can reduce cognitive overhead for platform teams already invested in the Kubernetes API. Terraform remains the most mature multi-cloud option with a large community and module registry, but these alternatives address specific pain points like language choice, native cloud integration, or Kubernetes alignment.

Frequently asked questions

Can I migrate my existing Terraform state to Pulumi or Crossplane?
Yes, both Pulumi and Crossplane offer migration guides and tools to import existing resources. Pulumi can adopt Terraform state and convert it to its own state format. Crossplane can take over management of existing resources through its managed resource adoption features, though the process may require manual mapping of resource IDs.
Is AWS CloudFormation free or does it have hidden costs?
AWS CloudFormation itself has no additional cost beyond the AWS resources it provisions. You pay only for the underlying services (like EC2 instances, S3 buckets, etc.) that the template creates. There are no per-stack or per-API-call fees from CloudFormation.
